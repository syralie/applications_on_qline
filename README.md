# Applications on Qline

We implement quantum communication protocols derived from Quantum Key Distribution (QKD) mechanisms, with a primary focus on the [Qline](https://veriqloud.com/qline/) architecture developed by VeriQloud.

# Methodology

To support application development on top of the Qline system, we designed a layered software stack that enables quantum communication protocols and provides a testbed for future applications.
The stack consists of three layers, all of which must be deployed by both communicating parties in order to run applications.

- Hardware Layer
At the hardware layer, users may choose between the physical Qline device and a hardware simulator.
The simulator reproduces the behavior of Qline in a one-to-one manner.
Throughout this paper, we refer to the hardware simulator as **hwsim**.

- Global Counter Layer
The Global Counter layer is responsible for maintaining synchronization between the communicating parties.
We refer to this layer as **GC**.


- Application Software Layer
The application software layer includes protocols built on top of the quantum communication.
Currently, we support two applications: quantum oblivious transfer and memory-less quantum tokens.


## Workflow

The **GC** software provides an interface between the hardware layer and the application layer. It receives start and stop commands from the application layer, triggers the hardware and handles the post selection over the measured quantum states through classical communication. 
At a high level, correlated raw data are generated by the hardware layer and writen into local FIFO buffers in a synchronized manner.
This design guarantees that application-layer software can continuously read correlated data on both sides.

We provide a control API to actively start or stop the key data stream from the application layer.
Conceptually, this API acts as a data-stream switch, controlled via a classical communication channel.


By decoupling data generation from application consumption, this design enables fine-grained control over resource usage and supports dynamic application behavior without compromising synchronization guaranties.


# Usage


## with hwsim as the Hardware Layer

- Clone and build [hw_sim](https://github.com/Veriqloud/hw_sim), obtain the executable **sim**.

- Clone and build [GC](https://github.com/Veriqloud/kiwi_hw_control/tree/master/gc), obtain the executable **gc_alice** and **gc_bob**. 

- Generate config files by [kiwi_hw_control/config/gen_config](https://github.com/Veriqloud/kiwi_hw_control//tree/master/config/gen_config)

- Run the hardware simulations and GC programs sequentially in the following order:
*sim_bob → sim_alice → gc_bob → gc_alice*, using the corresponding configuration files with argument *-c*. For debugging purposes, we recommend running each command in a separate terminal.

For example, place the configuration files in kiwi_hw_control/config/sim/. Then execute the generated binaries:

```bash
sim -c bob/sim.json
sim -c alice/sim.json
gc_bob -c bob/gc.json
gc_alice -c alice/GC.json
```


- At this stage, all components below the application layer are operational. Execute the protocol programs by following the instructions in the README of the corresponding protocol subfolder.

## with real Qline hardware as the Hardware Layer

- Initialize the Qline system. After the automatic initialization process completes, both the hardware layer and the GC layer should be activated and ready for operation.


- Execute the protocol programs by following the instructions provided in the README of the selected protocol subfolder.
